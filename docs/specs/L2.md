# L2 — Detailed Requirements

## L1-1: Link Command

### L2-1.1: CLI Options

The root command accepts `--workspace` (`-w`), `--library` (`-l`), and `--source` (`-s`) options, all required. These options are defined once and shared across all commands.

**Acceptance Criteria:**
- Invoking the CLI without required options produces a usage error.
- All three options are parsed and forwarded to the service layer.

### L2-1.2: Input Validation

The link command validates that the workspace path exists and contains `angular.json`, and that the library source path exists and contains a `package.json` with a matching library name.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when the library source path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.
- Returns exit code 1 when `package.json` is missing or has a mismatched name.
- No npm commands are invoked when validation fails.

### L2-1.3: npm link Execution

The link command invokes `INpmClient.LinkGlobalAsync` in the library source directory, then `INpmClient.LinkIntoWorkspaceAsync` in the workspace directory.

**Acceptance Criteria:**
- `LinkGlobalAsync` is invoked first in the library source directory.
- `LinkIntoWorkspaceAsync` is invoked second in the workspace directory.
- If the first call fails, the second is not executed and the exit code is propagated.
- If the second call fails, its exit code is propagated.
- Returns exit code 0 on success.

### L2-1.4: tsconfig.json Path Update

The link command delegates to `ITsConfigEditor.AddPaths` to add path mappings for the library.

**Acceptance Criteria:**
- `compilerOptions.paths` receives entries for `<library>` and `<library>/*` pointing to relative paths to the library source.
- If `tsconfig.json` does not exist, the command succeeds without error.
- If a present `tsconfig.json` cannot be parsed (including JSONC with comments and trailing commas), the command returns non-zero.
- JSONC content (single-line comments, trailing commas) is parsed successfully.
- Paths containing spaces are handled correctly.

## L1-2: Unlink Command

### L2-2.1: CLI Subcommand Registration

The `unlink` subcommand accepts the same shared `--workspace`, `--library`, and `--source` options as the link command, all required.

**Acceptance Criteria:**
- `npm-link unlink -w <path> -l <name> -s <path>` is a valid invocation.
- Omitting any required option produces a usage error.

### L2-2.2: Input Validation

The unlink command validates that the workspace path exists and contains `angular.json`, and that the library source path exists.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.
- Returns exit code 1 when the library source path does not exist.
- No npm commands are invoked when validation fails.

### L2-2.3: npm unlink Execution

The unlink command invokes `INpmClient.UnlinkFromWorkspaceAsync` in the workspace directory, then `INpmClient.UnlinkGlobalAsync` in the library source directory.

**Acceptance Criteria:**
- `UnlinkFromWorkspaceAsync` is invoked first in the workspace directory.
- `UnlinkGlobalAsync` is invoked second in the library source directory.
- If the first call fails, the second is not executed and the exit code is propagated.
- If the second call fails, its exit code is propagated.
- Returns exit code 0 on success.

### L2-2.4: tsconfig.json Path Removal

The unlink command delegates to `ITsConfigEditor.RemovePaths` to remove the library's path mappings.

**Acceptance Criteria:**
- The `<library>` and `<library>/*` keys are removed from `compilerOptions.paths`.
- Other existing path entries are preserved.
- If `tsconfig.json` does not exist, the command succeeds without error.
- If a present `tsconfig.json` cannot be parsed, the command returns non-zero.
- JSONC content (comments, trailing commas) is parsed successfully.

## L1-3: Verify Command

### L2-3.1: CLI Subcommand Registration

The `verify` subcommand accepts the same shared `--workspace`, `--library`, and `--source` options, all required.

**Acceptance Criteria:**
- `npm-link verify -w <path> -l <name> -s <path>` is a valid invocation.
- Omitting any required option produces a usage error.

### L2-3.2: Input Validation

The verify command validates that the workspace path exists and contains `angular.json`.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.

### L2-3.3: Symlink Verification

The verify command checks that the library exists in the workspace's `node_modules` as a symbolic link pointing to the library source path.

**Acceptance Criteria:**
- Reports PASS when `node_modules/<library>` is a symlink resolving to the library source path.
- Reports FAIL when `node_modules/<library>` does not exist.
- Reports FAIL when `node_modules/<library>` exists but is not a symlink.
- Reports FAIL when `node_modules/<library>` is a symlink pointing to a different target.
- Scoped packages (e.g. `@my-org/my-lib`) are resolved correctly under `node_modules`.

### L2-3.4: tsconfig.json Path Verification

The verify command delegates to `ITsConfigEditor.VerifyPaths` to check both the presence and correctness of path mapping values.

**Acceptance Criteria:**
- Reports PASS when both `<library>` and `<library>/*` keys exist with values matching the expected relative paths.
- Reports FAIL when `tsconfig.json` does not exist.
- Reports FAIL when either key is missing.
- Reports FAIL when the exact mapping exists but its value points to the wrong path.
- Reports FAIL when the wildcard mapping exists but its value points to the wrong path.

### L2-3.5: Aggregate Result

The verify command returns a structured `OperationResult` with a PASS/FAIL message per check and an appropriate exit code.

**Acceptance Criteria:**
- Returns exit code 0 when all checks pass.
- Returns exit code 1 when any check fails.
- Each check produces a distinct status message.

## L1-4: Architecture

### L2-4.1: Dependency Injection

`Program.cs` uses `Host.CreateApplicationBuilder()` as the composition root. All services are registered in DI and resolved by command handlers.

**Acceptance Criteria:**
- `INpmClient`, `ITsConfigEditor`, and `INpmLinkService` are registered in DI.
- Command handlers resolve services via `IServiceProvider` — no direct construction of implementations.

### L2-4.2: INpmClient Abstraction

`INpmClient` provides typed methods (`LinkGlobalAsync`, `LinkIntoWorkspaceAsync`, `UnlinkFromWorkspaceAsync`, `UnlinkGlobalAsync`) backed by `NpmClient`, which uses `ProcessStartInfo.ArgumentList`.

**Acceptance Criteria:**
- No command path relies on `cmd /c`.
- `npm.cmd` is invoked on Windows; `npm` on non-Windows.
- Scoped package names are passed as individual arguments, not interpolated into a shell string.

### L2-4.3: ITsConfigEditor Abstraction

`ITsConfigEditor` encapsulates all tsconfig.json read/write behavior with JSONC support (`JsonCommentHandling.Skip`, `AllowTrailingCommas = true`).

**Acceptance Criteria:**
- `ITsConfigEditor` exposes `AddPaths`, `RemovePaths`, and `VerifyPaths`.
- `TsConfigEditor` handles JSONC parsing, relative path calculation, and file IO.
- `NpmLinkService` contains no direct tsconfig file access.

### L2-4.4: Structured Results

Application methods return `OperationResult` (exit code + messages). Console output is performed by the CLI layer, not the service layer.

**Acceptance Criteria:**
- `OperationResult` record exists with `ExitCode` and `Messages` properties.
- `LinkAsync`, `UnlinkAsync`, and `VerifyAsync` return `Task<OperationResult>`.
- `NpmLinkService` contains no `Console.Write` calls.

### L2-4.5: Clean Build

All package versions resolve without warnings.

**Acceptance Criteria:**
- `dotnet restore` produces no NU1603 warnings.
- `dotnet test` runs cleanly.
