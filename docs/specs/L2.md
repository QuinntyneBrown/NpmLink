# L2 — Detailed Requirements

## L1-1: Link Command

### L2-1.1: CLI Options for Link

The link command (root command) shall accept `--workspace` (`-w`), `--library` (`-l`), and `--source` (`-s`) options, all required.

**Acceptance Criteria:**
- Invoking the CLI without required options produces a usage error.
- All three options are parsed and forwarded to the link service.

### L2-1.2: Input Validation for Link

The link command shall validate that the workspace path exists and contains `angular.json`, and that the library source path exists and contains a `package.json` with a matching library name.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when the library source path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.
- Returns exit code 1 when `package.json` is missing or has a mismatched name.
- No npm commands are invoked when validation fails.

### L2-1.3: npm link Execution

The link command shall run `npm link` in the library source directory, then `npm link <library>` in the workspace directory.

**Acceptance Criteria:**
- `npm link` is invoked first in the library source directory.
- `npm link <library>` is invoked second in the workspace directory.
- If the first command fails, the second is not executed and the exit code is propagated.
- If the second command fails, its exit code is propagated.
- Returns exit code 0 on success.

### L2-1.4: tsconfig.json Path Update

The link command shall add path mappings for the library in the workspace's `tsconfig.json`.

**Acceptance Criteria:**
- `compilerOptions.paths` receives entries for `<library>` and `<library>/*` pointing to relative paths to the library source.
- If `tsconfig.json` does not exist, the command still succeeds.

## L1-2: Unlink Command

### L2-2.1: CLI Subcommand Registration

The CLI shall register an `unlink` subcommand that accepts the same `--workspace` (`-w`), `--library` (`-l`), and `--source` (`-s`) options as the link command, all required.

**Acceptance Criteria:**
- `npm-link unlink -w <path> -l <name> -s <path>` is a valid invocation.
- Omitting any required option produces a usage error.

### L2-2.2: Input Validation for Unlink

The unlink command shall validate that the workspace path exists and contains `angular.json`, and that the library source path exists.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.
- Returns exit code 1 when the library source path does not exist.
- No npm commands are invoked when validation fails.

### L2-2.3: npm unlink Execution

The unlink command shall run `npm unlink <library>` in the workspace directory, then `npm unlink` in the library source directory.

**Acceptance Criteria:**
- `npm unlink <library>` is invoked first in the workspace directory.
- `npm unlink` is invoked second in the library source directory.
- If the first command fails, the second is not executed and the exit code is propagated.
- If the second command fails, its exit code is propagated.
- Returns exit code 0 on success.

### L2-2.4: tsconfig.json Path Removal

The unlink command shall remove the path mappings for the library from the workspace's `tsconfig.json`.

**Acceptance Criteria:**
- The `<library>` and `<library>/*` keys are removed from `compilerOptions.paths`.
- Other existing path entries are preserved.
- If `tsconfig.json` does not exist, the command still succeeds.
- If the paths object does not contain the library keys, no error occurs.

## L1-3: Verify Command

### L2-3.1: CLI Subcommand Registration for Verify

The CLI shall register a `verify` subcommand that accepts `--workspace` (`-w`), `--library` (`-l`), and `--source` (`-s`) options, all required.

**Acceptance Criteria:**
- `npm-link verify -w <path> -l <name> -s <path>` is a valid invocation.
- Omitting any required option produces a usage error.

### L2-3.2: Input Validation for Verify

The verify command shall validate that the workspace path exists and contains `angular.json`.

**Acceptance Criteria:**
- Returns exit code 1 when the workspace path does not exist.
- Returns exit code 1 when `angular.json` is missing from the workspace.

### L2-3.3: Symlink Verification

The verify command shall check that the library exists in the workspace's `node_modules` and is a symbolic link pointing to the library source path.

**Acceptance Criteria:**
- Reports success when `node_modules/<library>` is a symlink whose target resolves to the library source path.
- Reports failure when `node_modules/<library>` does not exist.
- Reports failure when `node_modules/<library>` exists but is not a symlink.
- Reports failure when `node_modules/<library>` is a symlink but points to a different target.
- For scoped packages (e.g. `@my-org/my-lib`), correctly checks `node_modules/@my-org/my-lib`.

### L2-3.4: tsconfig.json Path Verification

The verify command shall check that `tsconfig.json` contains path mappings for the library and that the mapping values point to the correct paths.

**Acceptance Criteria:**
- Reports success when `compilerOptions.paths` contains both `<library>` and `<library>/*` keys with correct values.
- Reports failure when `tsconfig.json` does not exist.
- Reports failure when path mappings for the library are missing.
- Reports failure when only one of the two expected keys is present.
- Reports failure when the exact mapping exists but points to the wrong path.
- Reports failure when the wildcard mapping exists but points to the wrong path.
- Reports success only when both keys exist and both values match expected paths.

### L2-3.5: Aggregate Result and Output

The verify command shall report the status of each check clearly and return an appropriate exit code.

**Acceptance Criteria:**
- Returns exit code 0 when all checks pass.
- Returns exit code 1 when any check fails.
- Outputs a clear pass/fail status message for each individual check (symlink and tsconfig).

## L1-4: JSONC-Safe tsconfig Handling

### L2-4.1: JSONC Parsing Support

All tsconfig.json parsing shall use `JsonCommentHandling.Skip` and `AllowTrailingCommas = true`.

**Acceptance Criteria:**
- A tsconfig.json with single-line comments (`//`) can be parsed and updated by `link`.
- A tsconfig.json with trailing commas can be parsed and updated by `link`.
- A tsconfig.json with comments and trailing commas can be parsed and updated by `unlink`.

### L2-4.2: Parse/Write Failure Handling

When a present tsconfig.json cannot be parsed or written, `link` and `unlink` shall return a non-zero exit code instead of logging a warning and returning success.

**Acceptance Criteria:**
- `link` returns non-zero when tsconfig.json contains invalid JSON content.
- `unlink` returns non-zero when tsconfig.json contains invalid JSON content.
- The error message indicates the tsconfig.json parse failure.

## L1-5: Typed npm Process Invocation

### L2-5.1: INpmClient Abstraction

An `INpmClient` abstraction shall provide explicit methods for npm operations: `LinkGlobalAsync`, `LinkIntoWorkspaceAsync`, `UnlinkFromWorkspaceAsync`, and `UnlinkGlobalAsync`.

**Acceptance Criteria:**
- `INpmClient` interface exists with the four methods.
- `NpmClient` implementation translates each method into a process invocation.
- Application code calls `INpmClient` methods instead of constructing shell strings.

### L2-5.2: Structured Process Arguments

Process execution shall use `ProcessStartInfo.ArgumentList` instead of building a single argument string. On Windows, invoke `npm.cmd` directly; on non-Windows, invoke `npm`.

**Acceptance Criteria:**
- No command path relies on `cmd /c`.
- `npm.cmd` is used on Windows, `npm` on non-Windows.
- Scoped package names (e.g. `@my-org/my-lib`) are passed as individual arguments, not interpolated into a string.

## L1-6: Dependency Injection and Separation of Concerns

### L2-6.1: ITsConfigEditor Abstraction

A dedicated `ITsConfigEditor` abstraction shall encapsulate all tsconfig.json read/write behavior.

**Acceptance Criteria:**
- `ITsConfigEditor` interface exists with methods for updating paths, removing paths, and verifying paths.
- `TsConfigEditor` implementation handles JSONC parsing, path calculation, and file IO.
- `NpmLinkService` delegates tsconfig operations to `ITsConfigEditor` instead of doing them inline.

### L2-6.2: OperationResult Structured Results

Application methods shall return a typed `OperationResult` (or equivalent) containing exit code and messages, rather than writing directly to Console and returning raw `int`.

**Acceptance Criteria:**
- `OperationResult` type exists with exit code and messages.
- `LinkAsync`, `UnlinkAsync`, and `VerifyAsync` return `OperationResult`.
- Console output is performed by the CLI layer based on the result, not by the service layer.

### L2-6.3: DI Composition Root

`Program.cs` shall use `Host.CreateApplicationBuilder()` to register and resolve services via DI.

**Acceptance Criteria:**
- Services are registered in DI (`INpmClient`, `ITsConfigEditor`, `INpmLinkService`).
- Command handlers resolve services from DI rather than constructing `new NpmLinkService(new ProcessRunner())`.
- Shared option definitions are consolidated and defined once.

**Acceptance Criteria:**
- Test where exact mapping exists but points to a different folder → verify returns 1.
- Test where wildcard mapping exists but points to a different folder → verify returns 1.

- Test for `link` using JSONC with comments and trailing commas → succeeds.
- Test for `unlink` using JSONC with comments and trailing commas → succeeds.

### L2-7.3: Unlink Source Path Validation Tests

**Acceptance Criteria:**
- `UnlinkAsync_MissingLibrarySourcePath_ReturnsOne` test exists and passes.
- Assertion that fake process runner recorded no invocations when source path is missing.

### L2-7.4: Paths with Spaces Tests

**Acceptance Criteria:**
- Test for workspace path containing spaces → operations succeed.
- Test for source path containing spaces → operations succeed.
